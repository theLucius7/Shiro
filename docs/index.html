Overview

项目概览

仓库定位

当前仓库更接近“站点前端源码”而不是“可直接复用的官方主题”。设计语言来自上游 Shiro，但实现细节已经明显偏向自用。

上游关系

仍基于 Innei/Shiro 和 Mix Space 生态，但不再追求与上游完全一致。出现差异时，以当前仓库代码为准。

主要能力

前台站点、轻后台、内容编辑、评论管理、实时事件、代理 API、S3 上传、项目级 Storybook，一并放在同一个仓库里维护。

Architecture

架构分层

前台站点

基于 Next.js App Router，主要页面位于 src/app/(app)。服务端先拉聚合数据，再把状态分发到 Jotai 和 TanStack Query。

轻后台

后台入口仍在 Next 内，但具体页面是借助 React Router 在 /dashboard 下二次挂载完成，代码集中在 src/routes 和 src/app/(dashboard)。

数据访问

统一通过 @shiro/fetch 和 @mx-space/api-client 做 API 请求，按 server/client 场景分开实现。

实时层

通过 SharedWorker + Socket.IO 管理连接和广播，尽量减少多标签页重复建连。

扩展能力

在 src/app/api 下维护了一批只为当前站点服务的 route，包含 GitHub、TMDB、Bangumi、音乐代理、Webhook、AI 摘要和 S3 上传。

组件预览

storybook/ 是单独的 Vite 工程，用来预览和回归组件。当前文档站会把它一起部署到 GitHub Pages 的 /storybook/ 子路径。

Structure

目录结构

关键目录

src/
  app/           Next.js 页面与 API routes
  components/    UI、布局和业务模块
  providers/     根级上下文与业务 provider
  queries/       React Query 查询定义
  routes/        轻后台的 React Router 页面
  socket/        SharedWorker + Socket.IO

storybook/       组件预览工程
packages/fetch/  server/client 分离的请求封装
scripts/         构建、打包、部署辅助脚本
docs/            GitHub Pages 项目文档源码

主要技术栈

  • Next.js 16 + React 18 + TypeScript
  • Tailwind CSS + 自定义样式层
  • Jotai
  • TanStack Query
  • Socket.IO
  • Mix Space API Client
  • 单独的 Vite Storybook 工程

Development

本地开发

环境要求

Node.js >= 20
pnpm@10.12.4

站点启动

cp .env.template .env
pnpm install
pnpm dev

默认端口是 2323。

组件预览

cd storybook
pnpm install
pnpm dev

常用命令

pnpm dev
pnpm dev:turbo
pnpm build
pnpm analyze
pnpm lint

Deploy

发布与部署

当前仓库存在两条主要发布思路: Docker 运行和 standalone 打包。二者都更偏向当前站点的部署习惯，而不是通用主题分发。

Docker

cp .env.template .env
docker compose up -d

对应文件是 Dockerfile 和 docker-compose.yml。

Standalone

sh ./ci-release-build.sh
sh ./standalone-bundle.sh

构建脚本围绕 .next/standalone 组织，生成可直接分发的压缩包。

Pages

GitHub Pages 说明

当前配置

文档站源码在 docs/，Pages 部署 workflow 在 .github/workflows/deploy-pages.yml。这个 workflow 会做三件事:

  1. 安装根仓库依赖并构建 storybook/
  2. 把项目文档和 Storybook 组装到同一个静态站点目录
  3. 通过 GitHub Actions 部署到 GitHub Pages

部署完成后，文档站在 Pages 根路径，Storybook 在 /storybook/ 子路径。

Notes

自用边界

不要把它当官方模板

这里的默认值、环境变量和某些交互都偏向当前站点。直接拿去二次部署前，先确认 API、 Gateway、存储和第三方服务约定是否对得上。

文档站的角色

这套 GitHub Pages 不是产品官网，而是仓库级文档入口。它更适合给维护者、协作者和后续 fork 使用。
